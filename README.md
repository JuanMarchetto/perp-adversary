# perp-adversary

An adversarial economic conformance harness for Solana perpetual-futures risk engines.

## What it does

`perp-adversary` drives a real perps risk engine through **multi-step adversarial campaigns** (matched trades, position-aware price cranks, funding accrual, source-credit-lien draws, margin withdrawal, liquidation, and quantity auto-deleverage across multiple accounts and slots) and checks an engine **economic invariant as a runtime oracle after every step**. When a step breaks the invariant, the failing campaign is reported with a one-command repro. Campaigns are driven into genuinely populated source-credit-lien state — not the all-zero default where the invariant holds trivially (see "v0 result") — and, as of v0.4, that state is **earned through real engine operations** (a funding-settled positive PnL drives a real lien draw) rather than seeded (see "v0.4 — earned-state campaigns").

The conformance oracles (v0–v0.4) and the v0.5 global vault-conservation oracle all **held**. As of **v0.6 the harness reports its first candidate RED**: funding settlement is **not value-conservative** for a fractional-basis position — it permanently destroys one quote atom of claimable value (`Σ capital + Σ pnl`) per settled slot to unattributable vault slack, with no spec-req#14 sink and no live engine check to catch it (see "v0.6"). It is non-extractive and sub-atom-per-event, so it is reported as a **candidate to triage**, not a confirmed exploit.

v0 targets [Percolator](https://github.com/aeyakovenko/percolator) at commit `71c9032`. It is an **independent harness that depends on Percolator (Apache-2.0); it does not fork or modify it.** It complements Percolator's existing per-instruction Kani proofs and single-path fuzzing by checking the invariant *across a whole campaign*, which bounded per-function model checking does not express.

## The O1 oracle (and why it is trustworthy)

v0 checks one invariant: **source-domain realizability** — the spec's core promise that *"the attacker cannot transform uncollectible A paper profit into global B purchasing power"* (spec.md v16.8.5, Requirement #2).

The oracle does not invent this invariant. It re-implements, field for field, Percolator's own per-account validator `SourceCreditLienAggregateProofV16::validate()` (`src/v16.rs:3060-3100`), re-running its six relationships (face decomposition, claim-bound, the `effective ≤ ceil(face/SCALE)` realizability cap, atom alignment, reservation exactness, impaired well-formedness) as an independent oracle after every observed step.

The oracle's arithmetic core is **proven sound with Kani** (`realizability_is_sound`): within the engine's documented operating range, if the oracle clears a domain then all six exact inequalities hold. So a reported finding can never be a harness arithmetic bug. The proof follows the engine's own bounded-proof discipline (small whole-atom magnitudes scaled by `BOUND_SCALE`, plus symbolic sub-scale remainders covering the ceiling and alignment boundaries).

## v0 result

Against the pinned engine (`71c9032`), the engine **held source-domain realizability on every campaign — and, critically, those campaigns now reach genuinely populated source-credit-lien state, so the result is not vacuous.**

Reaching that state took engine-faithful work. The O1 relationships hold trivially over an all-zero `DomainObs`, so a campaign only tests the oracle if it drives the engine to draw a **non-zero source-credit lien**. The engine draws one lien, in one place: `execute_trade`'s `create_initial_margin_source_lien_if_needed` (`src/v16.rs:10260`), and only when a **risk-increasing** trade is made by an account that already holds positive, source-attributed PnL backed by counterparty backing. The engine's design then gets in the way of reaching this through a price walk, *by intent*: `accrue_asset_to_not_atomic` advances the permissionlessly-cranked `effective_price` (`src/v16.rs:7894`) but deliberately does **not** touch the authenticated anchor `raw_oracle_target_price` (written only by the authenticating activation path, `src/v16.rs:4407`). That separation is the safety property — if the permissionless crank could move the authenticated reference, a price walk would self-authenticate. The resulting target/effective lag is a documented, Kani-asserted conservative interlock: it locks **risk-increasing** trades (`validate_trade_position_preflight`, `src/v16.rs:8557`) and is priced into margin as `target_effective_lag_loss_penalty` (`src/v16.rs:905`). So a pure price walk that creates the PnL also, by design, blocks the risk-increasing trade that would draw a lien against it. (We verified this is intended, not a defect, via reproduction + code path + the spec + the engine's own crank Kani proof.)

So the harness establishes the lien precondition **the way the engine's own conformance test does** (`v16_risk_increasing_trade_creates_source_credit_lien_for_im`, `tests/v16_spec_tests.rs:580`): a `SeedSourceClaim` action seeds the account's positive source-attributed PnL and the matching counterparty backing — the market side via the engine's **public** `add_source_positive_claim_bound_not_atomic` / `add_fresh_counterparty_backing_not_atomic` entrypoints, the per-account side exactly as the engine's `account_fixture` seeds it — and then asks the engine to **validate** the resulting state. A real risk-increasing trade at the no-lag activation price then makes the engine run its genuine lien-drawing logic. Nothing in the engine is stubbed or weakened.

On the now-populated campaigns the engine **held O1 at every step**:

- a proptest property over **256 randomized** lien-drawing campaigns (random seeded claims × position sizes, two accounts) — every campaign drew a real lien (observed `source_lien_effective_reserved` spanning ~1–200 atoms across the range, the realizability cap and the rejection path both exercised) — **no violation**;
- a populated **JELLY-archetype** campaign — the account accumulates a source claim, the opening trade draws a lien of 100 atoms, and a second lever attempt is correctly **rejected** (`LockActive`) so the lien stays capped at realizable backing — **no violation at any step**;
- an **anti-vacuity gate** (`tests/anti_vacuity.rs`) that fails loudly if any future change makes campaigns stop producing a non-zero `source_claim_liened_num` — turning vacuity into a permanent test failure.

This is a **conformance signal**, not a proof of correctness. It means: across the adversarial campaigns this harness drove — including states where the engine had actually drawn source-credit liens — the engine never left an accepted observable state that violated its own per-account source-credit lien-aggregate invariant. No candidate violation appeared; if one ever does, the proptest persists its scenario to `scenarios/realizability_candidate.json` and the run fails loudly (the oracle is never weakened to pass).

## v0.2 — the liquidation insurance-isolation oracle

The **liquidation attack class**: a liquidation must settle a bankrupt account's loss against *its own* domain's insurance backstop — it must never let the liquidated account's loss drain a **different** domain's shared insurance. This is an economic isolation guarantee between domains (e.g. liquidating an under-funded JELLY position must not silently spend BTC's insurance).

This is a **cross-step (delta) property**, and that is the whole point. After a liquidation the driver runs `validate_shape` + `validate_with_market`, so the post-liquidation state has already passed every *single-state* validator — re-checking any single state would be **vacuous** (the trap this class fell into twice). The isolation property only exists as a *change*: `insurance_domain_spent` is a running per-domain total, so isolation can only be observed by comparing each domain's value **before vs after** the liquidation against the step's reported `insurance_used`.

The oracle (`liquidation_insurance_isolation`, `src/oracles.rs`) mirrors, line for line, the engine's own `consume_domain_insurance_for_negative_pnl` (`src/v16.rs:5949`): a liquidation of an `asset` leg charges **only** `insurance_domain_index(asset, opposite_side(leg.side))` (`src/v16.rs:5955`) — a domain of *that asset* — and increments exactly that one domain's `insurance_domain_spent` by exactly the `insurance_used` it returns (`src/v16.rs:5974-5980`). Over the `(prev, cur)` observation deltas the oracle enforces, fail-closed: (I1) every domain's spend is **monotone**; (I2) **isolation** — only the liquidated asset's domains may increase (no cross-domain drain); (I3) **full accounting** — the total spend increase across all observed domains equals `insurance_used`. (I2)+(I3) together pin `insurance_used` to the liquidated domain's delta and force every other domain's delta to zero — exactly the engine's per-domain guarantee, as asserted by its own conformance test (`tests/v16_spec_tests.rs:342-348`) and bounded by its Kani proofs (`tests/proofs_v16.rs:2375` budget caps the spend; `:2410` reserved insurance is not double-spent; `:2449` an unfunded domain cannot drain shared insurance).

The oracle's core is **proven sound with Kani** (`liquidation_insurance_isolation_is_sound`, `tests/kani_liquidation.rs`): if it clears a liquidation step then no non-liquidated-asset domain's spend increased and `insurance_used` is fully accounted by the liquidated asset's spend delta. The proof models the engine's four-domain layout (two assets × long/short) with symbolic pre/post spends, a symbolic liquidated asset, and a symbolic `insurance_used` — the same bounded discipline as the O1 / cross-link proofs.

**Non-vacuity, the strong way.** The isolation oracle is meaningful even on the no-insurance path (it asserts *no* domain's insurance was spent). But the stronger test is a liquidation where insurance **is** spent for the correct domain and **none** for any other. The harness reaches that engine-accepted state: `funded_liquidation_campaign` funds the long leg's bankruptcy domain (`insurance_domain_budget_short`) exactly where the engine's own `proof_v16_view_domain_budget_caps_bankruptcy_insurance_spend` (`tests/proofs_v16.rs:2375`) funds it, then liquidates through the **public** entrypoint. Result: `insurance_used = 5`, the asset's short-side `insurance_domain_spent` rises 0 → 5, and *every other domain stays at 0* (`tests/liquidation.rs` gate). Nothing in the engine is stubbed or weakened.

**v0.2 result.** Against the pinned engine (`71c9032`), the engine **held all three oracles across both liquidation campaigns** (the residual/no-insurance path and the funded/insurance-spent path): O1 and the v0.1 cross-link held at every observed step (conformance under liquidation stress), and the v0.2 isolation oracle held on every consecutive step (`tests/liquidation_conformance.rs`). No candidate appeared; if one ever does, the scenario is persisted to `scenarios/liquidation_candidate.json` and the run fails loudly (the oracle is never weakened to pass).

## v0.3 — the ADL exact-accounting oracle

The **ADL (auto-deleverage) attack class**: when a bankrupt account's residual close has finalized, the engine deleverages the profitable counterparty by `apply_quantity_adl_after_residual_for_account_not_atomic` (`src/v16.rs:9479`). That close must be **exactly accounted** in the account's close-progress ledger — the quantity recorded as auto-deleveraged must equal the quantity actually closed, neither over- nor under-credited. An adversary who could make the ledger record *less* applied-ADL than was closed (or *more*) would corrupt the close-progress accounting that gates every subsequent crank on that account.

This is a **cross-step (delta) property**, and that is again the whole point. After an ADL step the driver runs `validate_shape` + `validate_with_market`, so the post-ADL state has already passed every *single-state* validator; the seeded finalized-close ledger's residual equation is itself engine-validated (the engine's own `proof_v16_close_progress_ledger_residual_equation_is_enforced`, `tests/proofs_v16.rs:1716`). Re-checking the residual equation or any single ledger/leg shape would therefore be **vacuous** (the trap this class must avoid — third time). The accounting property only exists as a *change*: `close_progress.quantity_adl_applied_q` is a running ledger total, so exact accounting can only be observed by comparing its value **before vs after** the ADL against the step's reported `closed_q`.

The oracle (`adl_accounting`, `src/oracles.rs`) mirrors the engine's own `advance_close_progress_quantity_adl` (`src/v16.rs:9510-9536`): the ADL closes `out.closed_q == close_q` of the leg (`apply_quantity_adl_after_residual_internal`, `src/v16.rs:9676`) with `close_q > 0` required (`src/v16.rs:9609`/`9522`), then sets `ledger.quantity_adl_applied_q = out.closed_q` (`src/v16.rs:9533`) — but **only** after asserting the prior ledger value was `0` (`src/v16.rs:9530`, else `LockActive`). So across one ApplyAdl step the ledger rises from 0 by exactly `closed_q`. Over the `(prev, cur)` observation deltas the oracle enforces, fail-closed: (E1) **exact accounting** — `quantity_adl_applied_q` rose by exactly `closed_q`; (E2) **non-vacuous close** — `closed_q > 0`; (E3) **ledger coherence** — the account's observed ledger value equals the outcome's reported post-ledger value (two views of the same engine field). The acted-on account is taken from the step's own `ApplyAdl` action, and its pre-ADL baseline is read from the matching account in `prev`.

The oracle's core is **proven sound with Kani** (`adl_accounting_is_sound`, `tests/kani_adl.rs`): if it clears an ApplyAdl step then the ledger rose by exactly `closed_q` (no backward move, no over/under-credit), `closed_q > 0`, and the two ledger views coincide. The proof draws the pre/post ledger values and the outcome's `closed_q` / reported value as independent symbolic quantities over a tiny range spanning increase / no-change / decrease and the zero-close case — the same bounded discipline as the O1 / cross-link / isolation proofs (the accounting law is scale-invariant).

**Non-vacuity.** The oracle only *means* something if the harness actually fires a real ADL: `adl_campaign` seeds a finalized-close, ADL-eligible state (an engine-validated finalized close-progress ledger plus an active bankrupt-side leg, exactly the gates the engine's ADL entrypoint checks) and then applies a quantity-ADL that closes `POS_SCALE`, advancing the ledger 0 → `POS_SCALE`. The `tests/adl.rs` gate fails loudly if the campaign ever stops firing a real ADL (`closed_q > 0` with the ledger advancing by exactly that amount). Nothing in the engine is stubbed or weakened.

**v0.3 result.** Against the pinned engine (`71c9032`), the engine **held all four oracles across the ADL campaign**: O1 and the v0.1 cross-link held at every observed step (conformance under ADL stress), the v0.2 isolation oracle held on every consecutive step (a no-op on the non-liquidation steps), and the v0.3 accounting oracle held on the ApplyAdl step (`tests/adl_conformance.rs`). No candidate appeared; if one ever does, the scenario is persisted to `scenarios/adl_candidate.json` and the run fails loudly (the oracle is never weakened to pass).

## v0.4 — earned-state campaigns

Through v0.3 every attack class established its precondition by **direct field writes** the engine then validated (the `SeedSourceClaim` / `SeedUnderwaterPosition` / `SeedFinalizedClose` actions): we wrote `account.pnl`, the per-account `source_claim_bound_num`, and the group PnL totals, then asked the engine to accept the result. That is honest — the engine validates the state it then operates on — but it only ever observes states the engine **accepts when handed to it**. The v0.4 thesis is sharper: **drive the engine through REAL operation sequences so the precondition is *reached* by engine logic**, because composition bugs live in *reachable transitions*. A clean conformance result on an earned transition is a stronger signal than the same result on a seeded one; a real oracle violation on an earned transition would be a genuine finding.

`earned_lien_campaign` (`src/driver.rs`) re-derives the **source-credit-lien** precondition end-to-end, replacing the direct-field parts of `SeedSourceClaim`. What is now **earned by the engine**, with no direct writes:

- **the open leg** — a real matched position from `execute_trade_with_fee_in_place_not_atomic`, not a hand-built `PortfolioLegV16Account`;
- **the source-attributed positive PnL** — settled by the engine from a favorable **funding** accrual. Each `AccrueFunding` crank routes through `permissionless_crank_not_atomic`'s `Refresh` (refresh-then-accrue): the refresh settles the open long leg's now-stale favorable K/F delta via `settle_leg_kf_effects_at_slot` → `apply_signed_kf_delta_to_pnl` → `set_account_pnl_with_source` (`src/v16.rs:7197`/`6627`), and `set_account_pnl_inner` (`src/v16.rs:6673`) is what raises **both** `account.pnl` **and** the per-account `source_claim_bound_num` (and the market `positive_claim_bound_num`) — all through engine logic. A **negative** `funding_rate_e9` makes `f_long_num` rise, which a long leg books as a gain (`src/v16.rs:7892`/`12565`). Critically, the funding crank keeps **`effective_price == raw_oracle_target_price`** (no price walk → no target/effective lag), so the favorable funding raises PnL **without** the lag that would otherwise lock the risk-increasing trade (`src/v16.rs:8557`) — exactly the interlock that blocked the v0 price-walk path;
- **the lien** — drawn by the engine inside `execute_trade`'s `create_initial_margin_source_lien_if_needed` (`src/v16.rs:10261`) on a final **risk-increasing add**, against the earned, funding-attributed positive PnL.

What is **still provider-posted** (a legitimate public operation, kept by design, not a workaround): the **market-side** source-credit claim + counterparty backing, via the engine's public provider entrypoints `add_source_positive_claim_bound_not_atomic` (`src/v16.rs:4909`) and `add_fresh_counterparty_backing_not_atomic` (`src/v16.rs:5058`) — the `ProviderPostClaim` action. This is what a backing provider does on-chain to post credit; the provider op has **no internal callers** (`src/v16.rs:4909`), so this market-side claim can only be established by a provider posting credit. It writes no per-account field and no group-header total. (A long leg's favorable funding is attributed to the **short** source-credit domain — `opposite_side(Long)`, `src/v16.rs:7196` — so the provider backs that domain and the lien draws there.)

Reaching the earned lien needs a **funding-enabled market config**: the canonical public config sets `max_abs_funding_e9_per_slot = 0`, so `accrue_asset_to_not_atomic` rejects any funding (`InvalidConfig`, `src/v16.rs:7833`). The earned campaign selects a config with `max_abs_funding_e9_per_slot = 10_000` (and `max_price_move_bps_per_slot = 9_999` so the engine's **exact solvency envelope** still accepts it on its branch-2 path, `src/v16.rs:1486`). That config passes the engine's own `validate_public_user_fund` (`src/v16.rs:1676`), so it is a **legitimate funded market**, not a weakened one — and only the earned campaign uses it; every seeded campaign keeps the funding-off public config.

**Non-vacuity.** `tests/earned_lien.rs` fails loudly unless the long reaches **strictly positive PnL through funding** *and* a non-zero per-account `source_claim_bound_num` *and* the engine draws a non-zero `source_claim_liened_num` / `source_lien_effective_reserved` — all without the seeded `account.pnl` / `source_claim_bound_num` writes. In the pinned engine the campaign earns ~400 atoms of source-attributed PnL over six funding cranks, then the risk-increasing add draws a **100-atom effective lien** against it. The final state passes the engine's own `validate_shape` + `validate_with_market`.

**v0.4 result.** Against the pinned engine (`71c9032`), the engine **held all four oracles on the EARNED transitions** (`tests/earned_lien_conformance.rs`): O1 and the v0.1 cross-link held at every observed step (now over a lien the engine actually *drew* against earned, funding-attributed PnL), and the v0.2 isolation + v0.3 accounting delta oracles held on every consecutive step (no-ops on the earned campaign's non-liquidation/non-ADL steps). No candidate appeared; if one ever does, the scenario is persisted to `scenarios/earned_lien_candidate.json` and the run fails loudly (the oracle is never weakened to pass). The legacy `SeedSourceClaim` / `lien_creating_campaign` path is **kept alongside** the earned path (the v0/v0.1 vacuity gates depend on it).

## v0.5 — the global quote-value conservation oracle (the PIVOT to unchecked/emergent invariants)

Every oracle through v0.4 **mirrors a per-state or per-instruction validator the engine itself runs** on every accepted state (`SourceCreditLienAggregateProofV16::validate`, the close-progress ledger advance, …). Because the engine rejects any state that would break them, those oracles hold on every engine-accepted state *by construction* — they are **conformance** checks, and a green result is (correctly) vacuous as a bug signal. v0.5 is a deliberate **pivot**: it checks an **emergent property the engine does NOT enforce as a single global validator across a whole campaign** — *total real quote-atom value is conserved over an entire multi-instruction campaign, changing only by external flows*. A value **leak or mint** — quote atoms created or destroyed across an internal step beyond the external flow — is the most serious class of DeFi bug. So unlike the conformance oracles, **a green result here is a genuine, non-vacuous independent signal**, and a red result is a **candidate to be triaged** (real leak vs incomplete value model), never a confirmed bug on its own.

**The value model — `system_quote_value(state) == vault`.** Per spec.md req #13 (`spec.md:46`) conservation is over **quote atoms only**; "encumbrances, source-credit reservations, backing buckets, and liens are not value classes." Per req #14 (`spec.md:47`) rounding residue has explicit sinks. So **PnL, claims, liens, backing, and reservations are accounting of claims, not real value.** The authoritative quote-atom model is the engine's own stock reconciliation (`spec.md:1043`; `StockReconciliationProofV16::validate`, `src/v16.rs:3029-3041`), which proves the **total** real quote-atom balance is the **token vault**, and that vault equals the sum of all *partition* stock classes:

```text
token_vault == senior_capital_total  (== C_tot == Σ account capital)
             + insurance_capital
             + backing_provider_earnings
             + settlement_rounding_residue_total
             + unallocated_protocol_surplus
```

Therefore **`vault` (`src/v16.rs:4061`) IS `system_quote_value`** — the single source-of-truth total real quote-atom store; every other quote store is a *slice* of it. We deliberately do **not** sum the slices: `c_tot == Σ capital` (`src/v16.rs:1072`) is a *derived aggregate* of value already inside `vault`, and adding it (or `insurance`, or `backing_provider_earnings`) on top would **double-count**. **Excluded and why:** `payout_snapshot` (`src/v16.rs:4087`) is a *snapshot* of payout entitlement (claim accounting, not a separate store); `explicit_unallocated_loss_*` (`src/v16.rs:3709`) is *loss accounting* whose value class `ExplicitBackedLoss` already lives inside `vault`; the partition fields (`c_tot`, `insurance`, `backing_provider_earnings`) are surfaced only for a partition cross-check (`senior <= vault`, `src/v16.rs:4590`), which fails closed if the observed model is ever inconsistent. The engine's own per-instruction validator `TokenValueFlowProofV16::validate` (`src/v16.rs:2913-2942`) enforces the *local* version `vault_after − vault_before == external_in − external_out`; the v0.5 oracle **composes that across a whole campaign at the harness level** from independently observed post-step `vault` snapshots — no engine proof object is consulted, so it is genuinely an independent check, not a mirror.

**External flow in the harness.** The only value that crosses the instance boundary in any campaign is `Deposit` (external in) and `Withdraw` (external out); no campaign invokes `resolve_market` or `claim_resolved_payout_topup` (no such `Action`), so nothing leaves the system as an external payout or fee on any step. The fixture seed `seed_underwater_position` injects quote atoms straight into `vault`; the driver records that injection as **external in** (it *is* value entering the instance), so it too is an honest external flow, not a mint. Consequently **every internal step must net exactly zero vault change** (`Δvault == ext_in_delta − ext_out_delta`).

**v0.5 result.** Against the pinned engine (`71c9032`), **conservation HELD across every campaign**: `lien_creating`, `earned_lien`, `liquidation`, `funded_liquidation`, `adl`, JELLY, an explicit deposit/withdraw `external_flow` campaign, and the 256-case adversarial proptest property — the total real quote-atom balance changed by **exactly** the net external flow on **every** step (`tests/conservation_campaign.rs`, `tests/runner.rs`). The oracle is proven **non-vacuous** by a negative control that plants a 1-atom mint into a real internal step and confirms the oracle catches it. **No candidate appeared.** If one ever does, the offending scenario plus a field-by-field before/after breakdown of every quote-atom store is persisted to `scenarios/conservation_candidate.json` and the run fails loudly — it is a candidate to triage (real leak/mint vs an incomplete value model), **never** a confirmed bug, and **the oracle is never weakened to pass**. The arithmetic core `value_conservation_kind` is **Kani-proven** both predicate-correct (`Ok` iff balanced and the partition is consistent, with the breach direction — appeared / disappeared / inconsistent — named correctly) and **overflow-safe / fail-closed** (an addition at the u128 ceiling never wraps into a spurious `Ok`).

## v0.6 — funding claimable-value conservation (the FIRST RED: a candidate value leak)

Every oracle through v0.5 **held**. v0.6 checks another emergent, engine-unchecked invariant — and the engine **does not hold it**. This is the harness's first reported candidate red. As designed, a red is a **candidate to be triaged** (a real conservation/spec break vs an accepted policy), never a confirmed exploit on its own; the triage is below and stated honestly.

**The invariant.** Funding is economically a **zero-sum transfer**: what the paying leg loses, the receiving leg gains. So the total **claimable value** `M = Σ (account.capital + account.pnl)` over all accounts must be **invariant under any funding settlement** (a step with no external flow). This is *not* the v0.5 vault invariant — `vault` is untouched by funding, so v0.5 stays green; `M` is a distinct measure the engine never reconciles globally.

**The break — a floor/ceil asymmetry across two separate instructions.** The two legs of a matched position settle their funding K/F deltas in **separate** `permissionless_crank` calls (`settle_leg_kf_effects_at_slot`, `src/v16.rs:7179`):

- the **receiver** leg (`net > 0`, `src/v16.rs:7194-7197`) credits PnL by `floor_div_signed_conservative_i128(+x) = ⌊x⌋ = q` (truncates down, `src/wide_math.rs:1435`);
- the **payer** leg (`net < 0`, `src/v16.rs:7198-7208`) debits **capital** by `|floor_div_signed_conservative_i128(−x)| = ⌈x⌉ = q+1` (rounds away from zero, `src/wide_math.rs:1441`), via `reserve_new_capital_backed_loss_for_source_domain_not_atomic` (`src/v16.rs:6998`, `c_tot -= q+1` at `:7028`).

When the per-leg magnitude `x = funding_delta·|basis| / (a_basis·POS_SCALE)` has a **nonzero remainder** — i.e. a **fractional basis** (`size_q` not a whole multiple of `POS_SCALE`) — `⌈x⌉ − ⌊x⌋ = 1`: the payer loses one **more** atom than the receiver gains. `M` falls by **exactly 1 quote atom per settled slot**. Because the two legs settle in different instructions, **no per-instruction `TokenValueFlowProof` spans both**, and each leg's own proof balances against the flat vault — so the asymmetry is never caught.

**Where the atom goes — nowhere the spec allows.** The destroyed atom leaves all tracked value classes (the vault is flat; `insurance`, `backing_provider_earnings`, `explicit_unallocated_loss` are unchanged) and is credited to **no sink** — a direct violation of **spec req #14** (`spec.md:47`): *"every quote-atom settlement/allocation residue caused by conservative rounding MUST be credited to `SettlementRoundingResidue` or `UnallocatedProtocolSurplus`"*, and of the atomicity rule (`spec.md:28`: *unattributed rounding residue* MUST roll the instruction back). It is **structurally undetectable** at runtime because:

- those two sink fields **have no storage** in the live `MarketGroupV16HeaderAccount` (`src/v16.rs:4057-4091`) — they exist only as fields of `StockReconciliationProofV16` (`src/v16.rs:3019-3025`); and
- `StockReconciliationProofV16` (`src/v16.rs:3019`/`3028`), the **only** object binding the partition equality `vault == Σ stock classes`, is **dead code** — **zero construction sites** in `src/` or `tests/` (the other three proof objects — `TokenValueFlowProofV16`, `ReservationEncumbranceProofV16`, `SourceCreditLienAggregateProofV16` — are referenced 25× in `src/v16.rs`).

**Reachable with no field-writes.** The campaign (`funding_leak_campaign`, `src/driver.rs`) uses only real public ops: `Deposit` + `Trade` (fractional `size_q`) + a `ProviderPostClaim` (the engine's public backing-provider entrypoint, no internal callers — `src/v16.rs:4909`, which backs the receiver domain so the long books its full floored gain every slot, no source-realizability haircut) + `AccrueFunding` cranks on the funding-enabled public config. The provider op injects **no** quote value into the measure (net external flow stays exactly the deposits), so the drift is genuinely engine-produced. The leak is **permanent, monotone, and unbounded** across epochs and fractional-basis legs.

**Lag vs leak (why the demonstration differences two lengths).** Settlement lags accrual by one refresh (refresh-then-accrue), so at any snapshot one slot of funding is "in flight" (payer debited, receiver not yet credited) — a **bounded, constant** shortfall offset that does **not** grow with campaign length. The **leak** is the part that grows by exactly 1 per added settled fractional slot. `tests/funding_conservation.rs` isolates the leak by differencing the claimable shortfall across lengths and across clean-vs-fractional bases: a **clean** basis shows a constant shortfall (`shortfall(10) == shortfall(30)`, just the lag, no leak); a **fractional** basis leaks **exactly one atom per added settled slot** (`shortfall(30) − shortfall(10) == extra settled slots`). So the signal can never be confused with the benign lag.

**Kani.** The oracle core `funding_conservation_kind` is proven **predicate-correct** (`Ok` iff claimable changes by exactly the net external flow, else the breach direction named) and **overflow-safe / fail-closed**. Separately, the **root cause** is proven against the engine's **real** `floor_div_signed_conservative_i128`: for one settlement of magnitude `x = x_num/d`, payer-debit `−` receiver-credit `== 1` iff `x_num % d ≠ 0` and `== 0` otherwise (`tests/kani_funding.rs`).

**v0.6 result (triaged honestly).** Against the pinned engine (`71c9032`), funding claimable-value conservation **does NOT hold for a fractional-basis position**: each settled slot permanently destroys exactly one quote atom of claimable value, to unattributable vault slack with no req#14 sink, undetected by any live engine check. **Honest limits:** the leak is **non-extractive** — the rounding is uniformly conservative (always *for* the protocol), so the slack accrues as over-collateralization and **no user gains**; it is one atom per settled fractional slot. So this is a **value-destruction / accounting-conformance / unattributed-residue** finding, not a drain exploit. A reviewer could argue the conservative direction is intended; the rebuttal is that req #14 mandates an **explicit** sink via a balanced flow proof and `spec.md:28` requires rollback of unattributed residue — neither happens, and the engine cannot even account for the accumulated slack because the sink fields and the reconciliation are absent. It is a genuine, reproducible *candidate* — exactly the "holds per-instruction, drifts across a campaign" target — to be triaged with the engine authors, not a confirmed bug.

## v0.7 — closing the realization gap: the PnL→capital firewall, now exercised (held)

After v0.6, a dedicated **extractive hunt** asked the sharper question: is there a path where a *user* withdraws/realizes **more than their entitlement** (the protocol loses value to a user), not just the protocol-favoring funding leak? The verdict was **no extraction found (~80-85% confidence the engine is uniformly conservative)** — every rounding site rounds *against* the user, and the source-credit backing partition is independently re-derived and proof-checked at multiple **live** layers (`SourceCreditLienAggregateProofV16`, `TokenValueFlowProofV16`, `ReservationEncumbranceProofV16`, all via `validate_with_market`). But the refutations leaned partly on **"unreachable"**: the harness had **never converted PnL to capital**, so the realization firewall was *argued* safe, never *exercised*.

`withdraw_not_atomic` (`v16.rs:11140`) pays only from `capital`; positive PnL is never directly withdrawable. The **only** PnL→capital path is `convert_released_pnl_to_capital_not_atomic` (`v16.rs:10821`, public) — and it had **zero callers** in the harness. v0.7 closes that gap: it wires `Action::ConvertReleasedPnl` + `Action::ReleaseLiens` and drives the firewall **end-to-end** (`convert_realization_campaign`): a long **earns funding PnL**, **flattens** its leg (so it has released PnL with no active exposure — past the `LockActive` exposure gate), **releases liens**, **re-certifies**, **converts** the released PnL into capital, and **withdraws** part of it.

**The security property, now tested not argued.** Convert does `capital += converted`, `pnl -= face_burn` with the **vault flat** (the `support_to_account_capital` flow proof, `v16.rs:10808`), after running the backing firewall (`create_and_consume_account_source_credit_for_effective_not_atomic`, `v16.rs:6185`, which fails closed to `LockActive` at `:6249` if realizable support cannot be covered by unliened backing). A **mint** (`converted > face_burn`) would raise claimable value (`Σ capital + Σ pnl`) with no external flow — exactly the `ClaimableValueCreated` (extractive) direction the **v0.6 oracle** detects. The tests (`tests/convert_realization.rs`) confirm:

- the firewall is **reached and non-vacuous** — the convert succeeds and realizes positive capital (`converted > 0`, PnL burned);
- it does **NOT mint** — claimable value is conserved **exactly** across the convert (`converted == face_burn`), so `funding_conservation_kind` returns `Ok`, never `ClaimableValueCreated`;
- the realized **withdrawal is fully backed** — global vault conservation (v0.5) holds on **every** step of the earn→flatten→convert→withdraw chain, so the withdrawn value is real counterparty value, not minted.

**v0.7 result.** The PnL→capital realization firewall — the last reachable surface where an extractive red could have hidden — **holds**: convert conserves claimable value exactly and the realized withdrawal is fully backed. The extractive conclusion is now *tested*, not argued. The residual unexercised seam (the liened-effective vs unliened-consume asymmetry, `v16.rs:5728` vs `:6204`, reachable only via convert against a flat account still carrying a reserved lien) is the natural next probe; the wiring to drive it now exists.

### Scope and honest limits

- **Four oracles.** O1 covers the per-account source-domain relationships (`src/v16.rs:3060`). The **v0.1 market-engine cross-link** oracle (`check_observation_market`) adds the outer realizability link `source_claim_bound_num ≤ positive_claim_bound_num` (per-account vs market-engine `positive_claim_bound_num`) plus the `market_id` binding — mirroring the engine's *composing* validator (`src/v16.rs:2143-2253`), i.e. the one relationship neither single-state validator enforces in isolation. (Re-checking the market-engine *single-state* validator would be vacuous: `try_to_runtime` runs it before the driver can observe the state, so any observable market-engine state has already passed it.) The **v0.2 liquidation insurance-isolation** oracle (`liquidation_insurance_isolation`) adds the *cross-step* domain-isolation guarantee above — the one property no single-state validator can express, because it lives in the delta. The **v0.3 ADL exact-accounting** oracle (`adl_accounting`) adds the *cross-step* close-progress accounting guarantee — that an auto-deleverage advances `quantity_adl_applied_q` by exactly the quantity it closed — again a delta no single-state validator can express (the post-ADL state already passed `validate_*`, and the residual equation is engine-validated). All four oracles are Kani-proven sound; the engine held all four across all campaigns. The **v0.5 global quote-value conservation** oracle (`value_conservation`) is different in kind: it does **not** mirror an engine validator — it checks the *emergent, engine-unchecked* invariant that total real quote-atom value (`vault`) is conserved across a *whole campaign*, so its green result is non-vacuous and a red result is a triage candidate (see the v0.5 section).
- **The lien precondition is now EARNED through funding (v0.4), not price-walked.** By design the authenticated oracle target is immutable after activation (the permissionless crank moves only `effective_price`), so a sustained *price* move deliberately locks risk-increase — there is no in-band path to draw a lien through a pure price *walk*. v0.4 instead earns the positive source-attributed PnL through **funding** at `effective_price == target` (no lag), so the engine raises `account.pnl` *and* the per-account `source_claim_bound_num` *itself* (`set_account_pnl_inner`, `src/v16.rs:6673`) and then draws the lien on a real risk-increasing add — the leg, the PnL, the claim, the lien, and the validation are all the engine's real code (`tests/earned_lien_conformance.rs`). Only the **market-side claim + backing** is still provider-posted, through the engine's public provider entrypoints, which is the legitimate on-chain action (no internal caller exists, `src/v16.rs:4909`). The legacy `SeedSourceClaim` seed path is kept alongside it for the v0/v0.1 vacuity gates.
- **Three attack classes.** Realizability (v0/v0.1, now also EARNED end-to-end in v0.4), liquidation insurance-isolation (v0.2), and ADL exact-accounting (v0.3). All are driven to genuinely non-vacuous engine-accepted states and checked cross-step; the engine held.
- **The funded-liquidation domain budget is seeded, not earned.** To make insurance genuinely spent (`insurance_used > 0`) the funded campaign sets `insurance_domain_budget_short` directly — exactly where the engine's own `proof_v16_view_domain_budget_caps_bankruptcy_insurance_spend` sets it — and then liquidates through the public entrypoint over an engine-validated state. The liquidation, the insurance spend, and the validation are all the engine's real code; only the pre-funded budget is seeded (no public budget-funding entrypoint exists to walk to it).
- **The ADL-eligible finalized-close state is seeded, not earned.** The ADL entrypoint gates on a `finalized && residual_remaining == 0` close-progress ledger plus an active bankrupt-side leg. `adl_campaign` constructs that ledger (satisfying the engine's residual equation `total_loss == progress`, finalized) and leg directly, then asks the engine to **validate** the state (`validate_shape` + `validate_with_market`) before the ADL runs. The ADL itself, the close arithmetic, the ledger advance, and the validation are all the engine's real code over a state it accepts; only the finalized-close precondition is seeded (the full residual-close cure path that earns it is the natural extension).
- **In-process, `_not_atomic` driving.** The harness calls the engine's building-block operations directly and observes after each; it does not yet exercise full atomic-instruction compositions or on-chain execution.
- **Bounded Kani proof**, matching the engine's own proof sizing — not an unbounded u128 proof (intractable for the model checker, as it is for the engine's own proofs). The bounded model does not exercise the oracle's overflow fail-closed arms, which only ever add violations, so soundness is unaffected.
- Pinned to engine SHA `71c9032`; the engine moves fast, so re-pin deliberately.

- **v0.4 (done)** — **earned-state** lien campaign: reach the source-credit-lien precondition through real engine ops (matched `execute_trade` leg + source-attributed positive PnL settled from **funding** at `effective_price == target`, no price walk) instead of seeding `account.pnl` / `source_claim_bound_num`; only the market-side claim/backing stays provider-posted via the public entrypoints. All four oracles run over the earned transitions; the engine held all four.
- **v0.5 (done)** — the **pivot to unchecked/emergent invariants**: a **global quote-value conservation** oracle (`value_conservation`) over `system.vault` — the engine's `StockReconciliationProofV16.token_vault` — checking that the total real quote-atom balance changes only by net external flow across a *whole campaign*, a property the engine enforces only *per instruction*, not globally. Kani-proven predicate-correct + overflow-safe, run as a cross-step delta over all five driver campaigns + JELLY + an external-flow campaign + the proptest property, with a non-vacuity negative control. **Conservation held everywhere**; no candidate appeared.
- **v0.6 (done)** — the **first RED**: a **funding claimable-value conservation** oracle (`funding_conservation_kind` + `claimable_shortfall`) over `Σ (capital + pnl)`. Reachable with no field-writes; permanent, monotone, unbounded; invisible to v0.5 (vault flat). Kani-proven predicate-correct + overflow-safe, with the root-cause floor/ceil asymmetry proven against the engine's real `floor_div_signed_conservative_i128`. **Non-extractive** (protocol-favoring), so reported as a candidate to triage, not a confirmed exploit.

The last reachable extraction seam **holds** — tested, not argued.
